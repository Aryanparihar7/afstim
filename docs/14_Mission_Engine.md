# Purpose

The Mission Engine is the core learning system of ShipLab.

It transforms educational content into structured, practical software engineering tasks.

Every mission should contribute directly toward building a production-ready application.

The Mission Engine exists to create builders, not course completers.

---

# Mission Philosophy

Every mission must answer one question:

"What practical capability will the learner gain after completing this mission?"

If a mission does not create a measurable capability, it should not exist.

---

# Mission Principles

Every mission should be:

Practical

Focused

Measurable

Progressive

Verifiable

No mission should exist solely to deliver theory.

Theory supports implementation.

Implementation remains the objective.

---

# Mission Structure

Every mission follows the same structure.

Mission Objective

↓

Context

↓

Learning Material

↓

Implementation

↓

Validation

↓

Reflection

↓

Completion

Consistency reduces learning friction.

---

# Mission Lifecycle

Draft

↓

Published

↓

Assigned

↓

In Progress

↓

Validated

↓

Completed

↓

Archived (if required)

Only published missions may be assigned to learners.

---

# Mission Objective

Every mission begins with a clear outcome.

Example

"Deploy your FastAPI backend to Railway."

The objective should describe the result, not the topic.

---

# Context

Before implementation, the learner should understand:

Why this task matters.

Where it fits within the application.

What they will accomplish.

Context should remain concise and practical.

---

# Learning Material

Learning material should prepare the learner for implementation.

It may include:

Short explanations

Diagrams

Code snippets

References

External documentation (where appropriate)

Avoid long-form lectures.

---

# Implementation

Implementation is the heart of every mission.

The learner performs the work inside the Workspace.

The platform encourages experimentation while maintaining clear objectives.

---

# Validation

Every mission defines explicit validation criteria.

Examples:

Application builds successfully.

API responds correctly.

Database connection is established.

Deployment succeeds.

Validation should be objective and repeatable.

---

# Reflection

Reflection converts experience into understanding.

Examples:

Why did this solution work?

What problem did this solve?

What would happen if this step were skipped?

Reflection should encourage independent thinking.

---

# Completion

A mission is complete only after:

Implementation

Validation

Reflection

All three are required.

Completion updates learner progress and reputation.

---

# Mission Difficulty

Version 1 supports three levels.

Beginner

Intermediate

Advanced

Difficulty should increase gradually.

Learners should rarely encounter sudden complexity.

---

# Mission Dependencies

Some missions depend on previous missions.

Prerequisites should remain explicit.

Learners may bypass prerequisites only through approved competency assessments.

---

# AI Integration

The AI Mentor supports missions by:

Explaining concepts.

Reviewing errors.

Providing contextual guidance.

Encouraging debugging.

The AI should not complete missions automatically.

---

# Progress Tracking

Mission progress is saved continuously.

Learners may leave and resume later without losing progress.

Progress updates automatically after successful validation.

---

# Mission Feedback

After every mission, learners may submit:

Difficulty feedback.

Bug reports.

Improvement suggestions.

This feedback supports continuous improvement.

---

# Mission Failure

Failure is treated as part of learning.

Validation failures should explain:

What failed.

Why it failed.

What to try next.

The learner should always have a path forward.

---

# Mission Consistency

Every mission should use the same layout.

Every mission should follow the same lifecycle.

Consistency allows learners to focus on solving problems instead of learning a new interface.

---

# Accessibility

Mission content should remain:

Readable

Keyboard accessible

Responsive

Clear

Learning should never depend on visual effects alone.

---

# Mission Analytics

Version 1 records:

Mission Started

Mission Completed

Average Completion Time

Validation Attempts

Common Failure Points

These metrics help improve future missions.

---

# Design Decisions

* Outcome-driven missions.
* Practical implementation first.
* Mandatory validation.
* Mandatory reflection.
* AI as a mentor.
* Consistent mission lifecycle.

---

# Known Risks

* Missions becoming too long.
* Validation becoming overly strict.
* AI providing excessive assistance.
* Learners skipping reflection mentally.

These risks should be reviewed through user feedback.

---
