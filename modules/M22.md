# M22 — Verification decision service

Track: Validation · Depends: M21 · Blocks: M25, M28 · Agent time: ~2h

## Goal
One place decides whether a mission is done. Nowhere else.

## Files you may touch
- `src/features/validation/services/verification-service.ts`
- `src/features/progress/services/progress-service.ts` (call it)

## Spec

```ts
evaluate(userId, missionSlug): Promise<{
  status: MissionStatus
  checks: Array<{ id, state, label, hint?, message? }>
  blockedBy: string[]        // human sentences
}>
```

**Verified when — all three, no exceptions:**
1. Every non-optional **local** check is `pass` (from `checkResults`)
2. Every non-optional **remote** check is `pass` (run now, server-side)
3. Reflection is non-empty (≥ 40 chars)

Any `PENDING` remote check → status stays `IN_PROGRESS`, `blockedBy: ["Still waiting on DNS."]`. **Never `FAIL`.**

On transition to `VERIFIED`:
- Set `verifiedAt`
- `track('mission_verified', { missionSlug, attempts })`
- If it's the last non-optional mission → `Enrollment.completedAt`, `track('journey_completed')`

**Idempotent.** Calling `evaluate` on an already-verified mission returns `VERIFIED` and **does not re-fire the event** or reset `verifiedAt`.

**Never downgrades.** Once `VERIFIED`, always `VERIFIED` — even if the learner's site later goes down. They did the thing. Taking it away would be cruel and would also mean their journey breaks when their free tier expires.

**This is the only function in the codebase that may write `status: VERIFIED`.** No route handler, no component, no admin action. If you find `VERIFIED` assigned anywhere else, that's the bug.

## Out of scope

1. Pass all checks, leave reflection empty → not verified, and it **tells you** reflection is missing. ☐
2. Write a reflection → verified. ☐
3. Take your site offline, reload the mission → **still verified**. ☐
4. `Event` has exactly one `mission_verified`. ☐
