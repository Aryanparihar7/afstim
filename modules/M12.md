## Spec

**This is the answer to "dynamic or static": static, file-based, version-controlled.**

No CMS. No mission table. A broken mission is a `git revert`, not a database incident. This decision deletes an entire admin content system from the build.

**1. Schemas** — Zod, matching `PLAN.md` Section 7 exactly.

Check types — **the closed set. Exactly seven. Adding an eighth requires founder approval:**
```
command_succeeds | file_exists | file_contains | http_ok        (local, CLI)
remote_http_ok   | remote_https_valid | remote_repo_exists      (server)
```

**2. `content-service.ts`**
```ts
getJourney(slug): Journey
listMissions(journeySlug): MissionMeta[]
getMission(slug): Mission          // frontmatter + compiled MDX
getLocalChecks(slug): Check[]      // CLI-runnable only
getRemoteChecks(slug): Check[]     // server-only
```
Read from disk at build time. Cache in memory. **Zero DB queries.**

**3. `scripts/validate-content.ts`** — runs in `prebuild`. Validates every file. On any failure:
```
✗ content/.../04-deploy-to-vercel.mdx
  checks[1].type: "remote_http_check" is not a valid check type.
  Valid types: command_succeeds, file_exists, file_contains, http_ok,
               remote_http_ok, remote_https_valid, remote_repo_exists
```
Then **exit 1**. A typo in a mission must break the build, not surprise a learner mid-mission.

**4. Two real missions** — 01 and 02 from `PLAN.md` Section 12 / Phase 9. Real prose, real checks. They prove the pipeline; M35 rewrites them properly from the founder's log.

## Out of scope
Any page. Any progress. Any check *execution* — this module only reads and validates definitions.

## Done when
- [ ] tsc, lint, build pass
- [ ] Break a frontmatter field on purpose → **build fails** with a readable message naming the file, the field, and the valid options
- [ ] `getLocalChecks` never returns a `remote_*` check
