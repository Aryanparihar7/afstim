## Spec

**`/register`** — Name (optional), Email, Password. Nothing else. `[Create account]`.
- Password rule: **10+ characters. That's the whole rule.** No symbol/uppercase/number theatre — it makes worse passwords and annoys people.
- Inline validation on blur, not on keystroke.

**`POST /api/v1/auth/register`**
- Zod validate
- Email already exists → **still return 200 with the same message.** Send a "someone tried to register with your email; here's a login link" mail instead. Never confirm an email exists. (Blueprint: *"Recovery should never expose account ownership information."*)
- Create user, `accessStatus: PENDING`, `emailVerified: null`
- Generate `VerificationToken`: 32 bytes hex, **24h expiry, single-use**
- Send via Resend
- `track('account_created')`

**Email** — plain, mono-free, one link, one sentence. Subject: *"Verify your email for Afstim"*. Body: *"Click to confirm this is you. The link works once and expires in 24 hours."*

**`/verify/[token]`** — validate → set `emailVerified`, **delete the token**, sign in, redirect `/dashboard`. Invalid/expired/used → *"This link has expired or was already used."* + `[Send a new one]`.

**`/verify`** — "Check your inbox" holding page + resend (max 3/hour/email).

## Out of scope
Login page (M10). Password reset — **not in v0.1**, you have 20 users, they can email you. Route protection (M11).

## Done when
- [ ] tsc, lint, build pass
- [ ] Token works exactly once
- [ ] Expired token gives a human sentence + a way forward
- [ ] Registering an existing email is indistinguishable from a new one

## Founder gate
1. Register with your real email. Mail arrives in **under 60 seconds**. ☐
2. Click the link → logged in. ☐
3. Click the same link again → *"already used"*, plus a button that helps. ☐
4. Register the same email again → same success message, no leak. ☐
5. Check the mail on your phone. It doesn't look like spam. ☐
