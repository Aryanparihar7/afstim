# M26 — Rate limits

One person can't cost you a month's budget in an afternoon.

## Spec

```prisma
model RateCounter {
  id        String   @id @default(cuid())
  key       String   @unique     // "check:<userId>:<YYYY-MM-DD-HH>"
  count     Int      @default(0)
  expiresAt DateTime
  @@index([expiresAt])
}
```
Migration: `add_rate_counter`

```ts
consume(key: string, limit: number, windowMinutes: number):
  Promise<{ allowed: boolean; remaining: number; resetAt: Date }>
```
Atomic upsert + increment. **No read-then-write** — two concurrent requests must not both pass on the last slot.

**Limits:**
| What | Limit |
|---|---|
| `POST /cli/results` | 30 / hour / user |
| `POST /recheck` | 1 / min / mission |
| `POST /feedback` | 3 / hour / IP (already in M07) |
| mentor | 25 / day / user — **M29, uses `User.aiCallsToday`, not this table** |

Over limit → 429: *"You've run a lot of checks. Try again in a few minutes."* Include `resetAt`. Never *"rate limited"* — that's our word, not theirs.

**Cleanup** — delete rows where `expiresAt < now()` on write. No cron.

**No Redis. No Upstash.** ADR-010 specifies distributed rate limiting for a product with zero users. It's cut. Postgres handles this at 10,000× our scale, and one fewer service is one fewer thing that can page you.

## Out of scope
Global WAF. IP reputation. Sliding windows — fixed hourly buckets are fine.
