# M20 — POST /api/v1/cli/results

Track: Validation · Depends: M19 · Blocks: M21, M26 · Agent time: ~3h

## Goal
What happened on the learner's machine reaches the website.

## Spec

**Request**
```json
{ "missionSlug": "deploy-to-vercel",
  "results": [ { "id": "build-passes", "state": "pass", "message": null } ] }
```

**Zod, strictly:**
- `missionSlug` must exist in content **and** be the learner's current unlocked mission → else 409 *"That isn't your current mission."*
- Every `id` must exist in that mission's local checks → else 400
- Every `id` must be a **local** check. A `remote_*` id here → 400. **The client does not get to report on server checks.**
- `state` ∈ `pass | fail`
- `message` ≤ 500 chars, or null

**Behaviour**
- Merge into `MissionProgress.checkResults` (JSON array of `{ id, state, message, at }`)
- Increment `attempts`, set `firstAttemptAt` if null
- `track('check_run', { missionSlug, passed, total })`
- Return the merged state including server checks (M21 fills those in)

**Cheating** — a learner can lie here. `curl` with `state: "pass"` works. **Accept it. Build no anti-cheat.** Server checks (M21) cover the claims that matter — the site is live, HTTPS works, the repo exists. Faking a local check only cheats you out of the thing you paid for. Anti-tamper costs weeks and protects nothing.

Do not add device fingerprinting, signed payloads, proof-of-work, or replay tokens. If you're tempted, reread this paragraph.

## Out of scope
