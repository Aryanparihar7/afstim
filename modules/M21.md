# M21 — Server check runners + SSRF guard

Track: Validation · Depends: M20 · Blocks: M22 · Agent time: ~4h
**⚠ The highest-risk module in the build. Read every line.**

## Goal
Afstim independently proves the learner's app is actually live on the actual internet.

## Files you may touch
- `src/features/validation/services/remote-check-service.ts`
- `src/features/validation/runners/remote-http-ok.ts`
- `src/features/validation/runners/remote-https-valid.ts`

## Spec

**Three runners. Server-side only. The learner can never influence the outcome.**

| Type | Behaviour |
|---|---|
| `remote_http_ok` | GET the submitted URL. Pass = `expectStatus`. 10s timeout, max 3 redirects (**re-guard every redirect target**). |
| `remote_https_valid` | TLS handshake. Pass = valid chain, not expired, not self-signed, hostname matches. |
| `remote_repo_exists` | GitHub public REST API. Pass = repo public + ≥1 commit. Unauthenticated. |

### The SSRF guard — this is the part that matters

**Our server is about to fetch a URL that a stranger typed.** That's a server-side request forgery hole, and it's how our infrastructure gets used to attack things.

`url-guard.ts`, applied **before every outbound fetch and again after every redirect**:

**Reject:**
- Scheme not `https:` (no `http:`, `file:`, `gopher:`, `ftp:`, `data:`)
- Hostname resolving to a private or reserved range: `10/8`, `172.16/12`, `192.168/16`, `127/8`, `169.254/16` (**this one is the AWS/GCP metadata endpoint — the classic**), `0.0.0.0/8`, `::1`, `fc00::/7`, `fe80::/10`
- Hostnames: `localhost`, `*.local`, `*.internal`, `metadata.google.internal`
- Any URL with credentials (`https://user:pass@…`)
- Port not 443
- Body larger than 1MB — read a capped stream, never `await res.text()` on an unknown host
- Response time over 10s

**Resolve DNS first, check the resolved IP, then connect.** Checking the hostname string is not enough — `evil.com` can resolve to `169.254.169.254`. If the runtime can't pin the connection to the checked IP, resolve → validate → connect by IP with the `Host` header set.

**On rejection:** `{ state: 'fail', message: 'That URL has to be a public https address.' }` — never explain *which* rule fired. Don't teach an attacker the ruleset.

### PENDING is not failure

DNS takes hours. If `remote_https_valid` fails with a DNS/connection error **and** the mission has a submission less than 24h old → return `PENDING`, not `FAIL`:

> *"Still propagating. This can take a few hours. Nothing's wrong — come back later."*

**A learner is never hard-blocked on a third party's clock.** That's a design rule, not an implementation detail.

### Recheck
`POST /missions/[slug]/recheck` — session auth, re-runs server checks, rate limited to 1/min/mission. Called by the page (M25) and by a button.

## Out of scope
Local checks. The `VERIFIED` decision (M22). Background jobs/cron — recheck on page load is enough at this scale.

## Done when

## Founder gate
1. Submit `http://localhost:3000` as your live URL → rejected. ☐
2. Submit `https://169.254.169.254` → rejected. ☐
3. Submit your real Vercel URL → passes. ☐
4. Submit a domain whose DNS you just changed → `PENDING` with the calm message, **not a red failure**. ☐
5. Ask the agent: *"can someone use Afstim to scan my hosting provider's internal network?"* Answer must be a specific no citing DNS resolution + IP checking. Vagueness → reject the module. ☐
