# M40b — Password reset flow

Track: Launch · Depends: M11 · Blocks: M41 · Agent time: ~3h

## Goal
A learner who forgets their password can get back in without emailing you.

## Why this exists
M10 deliberately deferred this for 20-user manual support. Found during
development: the person locked out can be the founder mid-build with no
one to email. Real users need a self-serve path before M36's five
humans arrive — nobody paying you should be permanently locked out.

## Spec
Same shape as email verification (M09/the OTP patch): generate a token,
hash it before storing, single-use, expiring (1 hour), emailed as a
link, same "don't reveal whether the email exists" response either way
as registration already does. Reuse existing rate-limit and email
patterns rather than inventing new ones.

## Out of scope
SMS reset. Security questions. Reusing the EmailOtp table (this is a
different token type with different expiry — keep it a separate model).

## Founder gate
1. Request a reset for a real account → email arrives with a link. ☐
2. Click it → set a new password → log in with the new one. ☐
3. Try the same link twice → second use rejected. ☐
4. Request reset for an email that doesn't exist → identical response,
   no leak. ☐
5. Old password no longer works after reset. ☐
